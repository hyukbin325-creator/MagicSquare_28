# Interactive Prompt Transcript

이 문서는 현재 세션에서 생성/가공된 프롬프트를 실행 가능한 형태로 순서대로 정리한 transcript이다.

---

## Turn 1 - 사용자 요청

> 너는 Cursor.AI용 프롬프트 설계자다.  
> 나는 Magic Square 4x4 TDD 설계 문서를 만들기 위한 프롬프트가 필요하다.  
> 지금 설계 문서를 만들지 말고, 그 설계 문서를 만들 수 있는 실행용 프롬프트만 작성하라.

## Turn 1 - 생성된 실행용 프롬프트

```text
역할:
너는 시니어 소프트웨어 아키텍트이자 TDD 코치다.
목표는 "4x4 Magic Square 생성기"에 대한 TDD 중심 설계 문서를 작성하는 것이다.

중요 지시:
- 지금 단계에서 코드 구현은 하지 마라.
- 설계 문서만 작성하라.
- 문서는 한국어로 작성하라.
- 모호한 요구사항은 "가정(Assumptions)"으로 명시하라.
- 테스트 가능한 형태로만 요구사항을 기술하라.

프로젝트 맥락:
- 대상: 4x4 Magic Square
- 정의: 1~16의 정수를 중복 없이 사용하여 4x4 격자를 채우고, 각 행/열/두 대각선의 합이 모두 동일해야 한다.
- 목표 합(매직 상수): 34
- 산출물은 TDD 관점(테스트 우선)으로 구성되어야 한다.

문서 산출 형식(아래 섹션 순서 고정):
1) 문서 개요
2) 문제 정의 및 범위
3) 기능 요구사항(Functional Requirements)
4) 비기능 요구사항(Non-functional Requirements)
5) 도메인 규칙/불변조건(Invariants)
6) API/모듈 설계 초안
7) 테스트 전략(TDD Strategy)
8) 테스트 케이스 목록(우선순위 포함)
9) Red-Green-Refactor 반복 계획(스프린트/단계별)
10) 예외/실패 시나리오
11) 완료 정의(Definition of Done)
12) 리스크 및 대응
13) 추후 확장 포인트(예: NxN 일반화)

각 섹션 작성 규칙:
- "문서 개요": 목적, 독자, 산출 범위를 5~8줄로 요약.
- "문제 정의 및 범위": 포함/비포함 항목을 표로 정리.
- "기능 요구사항": FR-01, FR-02 식별자 부여, 검증 가능한 수용 기준(acceptance criteria) 포함.
- "비기능 요구사항": 성능, 재현성, 결정성(determinism), 가독성, 테스트 실행 시간 기준 제시.
- "도메인 규칙/불변조건": 수학적 제약을 명시하고 위반 시 기대 동작 설명.
- "API/모듈 설계 초안": 모듈 책임, 입력/출력 계약, 오류 계약을 포함. (언어 중립적으로)
- "테스트 전략": 단위/통합/속성 기반 테스트 여부와 이유를 기술.
- "테스트 케이스 목록":
  - 최소 20개 이상.
  - 정상/경계/실패/회귀 카테고리로 분류.
  - 각 케이스에 Given-When-Then 포함.
  - 우선순위(P0/P1/P2) 표기.
- "Red-Green-Refactor 반복 계획":
  - 최소 6회 반복 사이클 제시.
  - 각 사이클마다 실패 테스트 -> 최소 구현 -> 리팩터링 목표를 3줄 이내로 정리.
- "예외/실패 시나리오": 잘못된 입력, 중복 숫자, 범위 초과, 불가능 상태 처리 방식 포함.
- "완료 정의": 테스트 커버리지 목표, 필수 통과 테스트, 문서 완료 기준 체크리스트 포함.
- "리스크 및 대응": 기술 리스크 5개 이상 + 완화 전략.
- "추후 확장 포인트": 4x4 전용 설계를 NxN으로 확장할 때 변경 지점 분석.

출력 품질 기준:
- 섹션/항목 번호 체계를 일관되게 유지.
- 모든 요구사항은 테스트 항목과 추적 가능해야 함(요구사항 ID ↔ 테스트 케이스 ID 매핑 표 포함).
- 불필요한 장문 설명보다 실행 가능한 설계 정보를 우선.
- 마지막에 "즉시 구현 시작용 체크리스트(10개 내외)"를 추가.

최종 출력 형식:
- Markdown 문서로 출력.
- 표, 체크리스트, 번호 목록을 적극 활용.
- 코드 블록은 API 시그니처 예시가 필요할 때만 최소한으로 사용.
```

> Dual-Track UI + Logic TDD 및 Clean Architecture 설계 전문가 프레임과

> 그리고 모호성 금지/검증 가능성/표 및 체크리스트 요구사항을 포함한 상세 지시문.

## Turn 2 - 생성된 실행용 프롬프트(정제본)

```text
역할:
당신은 Dual-Track UI + Logic TDD 및 Clean Architecture 설계 전문가입니다.

프로젝트:
Magic Square (4x4) — TDD 연습용

목적:
알고리즘 난이도보다 “레이어 분리 + 계약 기반 테스트 + 리팩토링” 훈련을 목표로 설계 문서를 작성합니다.

절대 제약:
- 구현 코드는 작성하지 마십시오. (설계/계약/테스트/통합 계획만)
- UI는 실제 화면이 아니라 “입력/출력 경계(Boundary)”로 정의하십시오.
- Data Layer는 DB가 아니라 “저장/로드 인터페이스(메모리/파일 교체 가능)” 수준으로만 정의하십시오.
- 입력/출력 계약은 아래 명세를 엄격히 고정하십시오.
- 모호한 표현 금지: “적절히/충분히/가능하면” 같은 표현 사용 금지.
- 모든 규칙은 테스트로 검증 가능해야 합니다.
- 표/체크리스트를 적극 사용하십시오.

입력 계약(고정):
- 타입: 4x4 int[][]
- 0은 빈칸
- 빈칸은 정확히 2개
- 값 범위: 0 또는 1~16
- 0 제외 중복 금지

출력 계약(고정):
- 타입: int[6]
- 좌표는 1-index
- 반환 형식: [r1,c1,n1,r2,c2,n2]
- n1,n2는 두 누락 숫자
- (작은수→첫빈칸, 큰수→둘째빈칸) 조합이 마방진이면 그 순서를 반환
- 아니면 반대 순서를 반환

문서 작성 규칙:
- 구현 코드 금지 (의사코드도 금지)
- 메서드는 “시그니처 수준 설명”만 허용 (언어 불문)
- 모든 규칙/계약/테스트 항목은 ID를 부여하고 상호 추적 가능하게 작성
- Error 메시지 규칙은 정확한 문구 템플릿까지 명시
- 출력은 반드시 아래 구조와 동일한 헤더 체계를 유지

출력 형식 (반드시 이 구조로):
# 1) Logic Layer (Domain Layer) 설계
## 1.1 도메인 개념
- Entities / Value Objects / Domain Services 목록과 책임(SRP)
## 1.2 도메인 불변조건(Invariants)
- 행/열/대각선 합 일치, Magic Constant 등
## 1.3 핵심 유스케이스(도메인 관점)
- 빈칸 찾기, 누락 숫자 찾기, 마방진 판정, 두 조합 시도
## 1.4 Domain API(내부 계약)
- 메서드 시그니처 수준(코드 X) + 입력/출력/실패조건
## 1.5 Domain 단위 테스트 설계(RED 우선)
- 테스트 케이스 목록(정상/비정상/엣지)
- 각 테스트가 보호하는 invariant 명시

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)
## 2.1 사용자/호출자 관점 시나리오
- “행렬 입력 → 검증 → 결과 출력” 흐름
## 2.2 UI 계약(외부 계약)
- Input schema / Output schema / Error schema
## 2.3 UI 레벨 테스트(Contract-first, RED 우선)
- 잘못된 크기, 빈칸 개수 오류, 값 범위 오류, 중복 오류, 반환 포맷 검증
- Domain은 Mock으로 가정
## 2.4 UX/출력 규칙
- 에러 메시지 표준(정확한 문구 규칙까지)

# 3) Data Layer 설계 (Data Layer)
## 3.1 목적 정의
- “저장/로드”의 필요성과 범위(학습용)
## 3.2 인터페이스 계약
- 예: MatrixRepository.save/load (메서드 수준, 코드 X)
- 저장 대상: 입력 행렬, 실행 결과(선택)
## 3.3 구현 옵션 비교(메모리/파일)
- 옵션 A: InMemory / 옵션 B: File(JSON/CSV)
- 추천안 1개 선택 + 이유
## 3.4 Data 레이어 테스트
- 저장/로드 정합성, 예외(파일 없음/형식 오류), 불변조건(4x4 유지)

# 4) Integration & Verification (통합 및 검증)
## 4.1 통합 경로 정의
- UI → Application(선택) → Domain → Data 흐름(의존성 방향 포함)
## 4.2 통합 테스트 시나리오
- 정상 시나리오 2개 이상
- 실패 시나리오 3개 이상(입력 오류, 도메인 실패, 데이터 실패)
## 4.3 회귀 보호 규칙
- 기존 테스트 유지 정책
- 변경 금지 규칙(계약/출력 포맷)
## 4.4 커버리지 목표
- Domain Logic 95%+
- UI Boundary 85%+
- Data 80%+
## 4.5 Traceability Matrix (필수)
- Concept(Invariant) → Rule → Use Case → Contract → Test → Component

추가 출력 요구:
- 반드시 Markdown으로 작성
- 표 3개 이상 포함
- 체크리스트 2개 이상 포함
- 각 테스트 케이스는 최소 필드(ID, 목적, 입력, 기대결과, 보호 규칙) 포함
- 마지막에 “실행 준비 점검표(Go/No-Go)” 섹션을 추가하고 체크리스트로 마무리
```
