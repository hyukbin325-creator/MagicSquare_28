## STEP 1 — Observation (관찰)

### 1. 현재 우리가 해결하려는 상황은 무엇인가?

**관찰:**  
누군가는 **4×4 격자**에 **1부터 16까지의 정수를 각 칸에 한 번씩** 배치했을 때, **행·열·두 대각선의 합이 모두 같아지는 배치**가 존재하는지, 그리고 그런 배치를 **어떻게 찾거나 생성·검증할지**를 다루려 한다.

동시에 이것은 **수학적 조건(합 일치, 숫자 중복 없음)**과 **컴퓨터가 수행할 작업(탐색, 규칙 적용, 출력, 사용자 입력)** 사이에 **아직 명시되지 않은 간극**이 있는 상태이다.  
무엇을 입력으로 받고, 무엇을 출력·판정하는지, 실패 시 어떻게 보여줄지는 아직 관찰 범위 밖이다.

**요약:**  
*4×4에서 마방진 조건을 만족하는 배치를 다루는 문제를, 소프트웨어로 다루기 위한 출발점에 서 있다.*

### 2. 왜 4×4 마방진 문제를 다루는가?

| 관찰 가능한 이유 | 설명 |
|------------------|------|
| **규모가 적당함** | 3×3보다 제약이 풍부하고, 5×5 이상보다 탐색·표현·검증이 부담이 적음 |
| **조건이 명확함** | “합이 같다”, “1~n²를 한 번씩”처럼 검증 기준이 수식·절차로 옮기기 쉬움 |
| **알고리즘·구조 학습** | 백트래킹, 휴리스틱, 대칭·패턴, 행렬/인덱스 다루기 등 작은 완전문제로 연습하기 좋음 |
| **설계 연습** | 입력/출력, 검증 모듈, 생성 vs 탐색 분리 등 작은 시스템의 경계를 나누기 좋음 |
| **프로젝트 이름** | MagicSquare_XX는 4×4에 초점을 둔 학습·실습 저장소일 가능성 |

**아직 확정되지 않은 것:** 교과/과제 vs 취미 vs 포트폴리오, “하나만” vs “모든 해” vs “사용자 퍼즐(고정 칸)” 등.

### 3. 어떤 학습 또는 시스템 설계 맥락에서 등장하는가?

**학습 맥락**

- 이산수학·조합: 제약 만족, 대칭, 해의 개수
- 프로그래밍: 2차원 배열, 반복, 조건 분기, 함수 분리
- 알고리즘: 탐색, 가지치기, 복잡도 감각
- 테스트·검증: “올바른 배치”의 자동 판정 정의

**시스템 설계 맥락 (소규모)**

- 도메인: 격자 상태 + 마방진 불변식
- 역할 분리 후보: 배치 생성 / 합 검증 / UI·CLI / 설정(크기 고정 4)
- 품질 이슈: 잘못된 입력, 해 없음, 중복 해, 성능(4×4는 보통 무시 가능)

**빈 저장소가 시사하는 것:** 구현 전에 요구사항·경계·성공 기준을 고정하려는 **문제 정의 흐름**.

### 관찰 한 줄 정리

> **4×4 격자에 1~16을 중복 없이 놓았을 때 여러 줄·대각선의 합이 같아지는 배치**를 대상으로, 그 **존재·생성·검증·표현**을 컴퓨터 절차로 옮기기 전에, **무엇을 목표로 하는지**를 아직 밝히지 않은 초기 상황이다.

## STEP 2 — Why #1

### Q: 왜 마방진을 완성해야 하는가?

#### 가정한 답 (A)

> **4×4 격자에 1~16을 한 번씩 넣어, 모든 행·열·대각선의 합이 같게 맞춘 하나의 올바른 배치를 얻어야, 그 결과가 “마방진 조건을 만족한다”는 것을 눈으로·수치로 확인할 수 있기 때문이다.**

완성의 목적은 수학적 조건의 충족을 **구체적인 격자 상태로 고정**하고, 그 상태를 **검증·제시·학습·과제 제출**에 쓰는 것이다.

#### A에서 드러나는 불편함 · 구조적 문제

| # | 구조적 문제 | 불편함 |
|---|-------------|--------|
| 1 | **완성 = 유일한 정답?** vs **임의 유효 해?** | 서로 다른 유효 배치가 “틀렸다”로 오인될 수 있음 |
| 2 | **완성** vs **검증 가능한 상태** 모호 | 부분 채움·퍼즐 모드에서 역할이 꼬임 |
| 3 | **결과만** 강조 | 과정·중간 실패 피드백이 설계에 반영되지 않음 |
| 4 | **“해야 한다”** 출처 불명 | 과제·자기검증·도구·데모 요구가 한데 섞임 |
| 5 | **4×4 고정** | n×n 확장 시 “완성” 정의가 달라져 재정의 필요 |

#### Why #1 요약

- **표면:** 조건을 만족하는 4×4 배치를 끝까지 채운다.
- **깊이:** 그 배치로 “마방진 조건 충족”을 증명·확인한다.
- **남는 빈틈:** 성공 기준, 완성 vs 검증, 결과 vs 과정, “해야 함”의 주체, 범위(4×4 고정 여부).

## STEP 3 — Why #2

### Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?

#### 1. 반복 가능성

| 답 | 구조적 문제 |
|----|-------------|
| 동일 입력·규칙 → 동일 절차를 N번 실행; 실험·재현 | 재현만 있고 반복 **목적**이 없으면 빈 루프; 결정론 vs 다양성·채점 충돌 |

#### 2. 검증 자동화

| 답 | 구조적 문제 |
|----|-------------|
| 합·중복·범위·빈 칸을 일관되게 판정; 유효성은 “특정 행렬 일치”가 아님 | 검증만 있고 생성 없음; 잘못된 검증 규칙은 오류를 고착; 피드백 깊이 부족 가능 |

#### 3. 오류 방지

| 답 | 구조적 문제 |
|----|-------------|
| 중복·범위·부분만 확인 등 사람 실수를 절차로 차단 | 과도한 제약은 학습 방해; 구현자의 규칙 오류는 그대로; 4×4는 손이 더 빠를 수 있음 |

#### 4. 규칙 기반 사고 훈련

| 답 | 구조적 문제 |
|----|-------------|
| 말로 된 요구를 실행 가능한 규칙(불변식·종료·절차)으로 고정 | 규칙만 옮기고 이해 없음; 규칙 폭발; “존재하는 해” vs “찾는 해” 혼동 |

#### Why #2 종합

| 관점 | 한 줄 |
|------|------|
| 반복 가능성 | 판정·생성 절차를 언제든 다시 돌림 |
| 검증 자동화 | 조건 충족을 절차로 판정 |
| 오류 방지 | 불가능·불일치 상태를 일찍 차단 |
| 규칙 기반 사고 | 요구를 실행 가능한 규칙으로 고정 |

**통합:** 단순 계산은 “이번 한 판”, 프로그램은 “판정과 절차를 시스템으로 고정”. 반복·규칙 정의 없이는 다른 형태의 구조적 문제로만 바뀐다.

#### 다음 정의용 질문

1. 1차 산출물: 생성된 격자 / 검증 리포트 / 둘 다?
2. 반복: 사용자 요청 vs 대량 실험?
3. 오류 방지: 사용자 입력 차단 vs 내부 탐색만?
4. 규칙 훈련 대상: 작성자 vs 최종 사용자?

## STEP 4 — Why #3 (핵심 발견)

### Q: 왜 이 문제를 TDD 방식으로 설계하려 하는가?

4×4는 손으로도 가능해 TDD가 **필수**인 난제는 아니다. TDD 선택은 **판정권·불변식·계약**을 먼저 밖으로 꺼내겠다는 선언에 가깝다.

### 1. 무엇이 통제되어야 하는가?

| 통제 대상 | 통제하지 않으면 |
|-----------|-----------------|
| “맞다”의 정의 | 다른 유효 해 = 실패 오인 |
| 검증 규칙 | 부분만 맞아도 통과, 잘못된 규칙 고착 |
| 생성 vs 검증 경계 | 검증 버그가 생성에 묻힘 |
| 상태 전이·회귀 | 변경 시 어디가 깨졌는지 모름 |

**발견:** 통제의 중심은 UI·속도가 아니라 **판정권**이다.

### 2. 어떤 불변 조건이 존재하는가?

**값·칸:** 정의역, 칸마다 유일, 완성 시 1~16 각 1회  
**합 (4×4, 1~16):** 마법상수 34; 4행·4열·2대각선 각 합 = 34  
**부분 상태:** 유일·범위 유지; 완성 전에는 모든 선 합=34 필수 아님; 초과 시 불가 가지  
**메타:** 검증은 생성을 신뢰하지 않음; 테스트 fixture도 검증 통과 보장

### 3. 왜 입력/출력이 명확해야 하는가?

테스트는 **Given → When → Then**으로 스펙을 고정한다.

| 경계 | 입력 | 출력 |
|------|------|------|
| 검증 | 4×4 (또는 빈 칸 표기) | valid/invalid + 위반 종류 |
| 생성 | 없음/시드/방식 | 격자 또는 실패 |
| 부분 검증 | 일부 채움 | complete_valid / partial_ok / violated |

**발견:** 입출력 명확화는 테스트가 **실행 가능한 스펙**이 되는 조건이다. 실패 유형까지 포함하는 편이 STEP 2~3과 정합된다.

### Why #3 한 줄 발견

> TDD는 “테스트 기법”이 아니라, **마방진을 ‘그림’이 아니라 ‘불변식 + 명시적 계약’으로 다루겠다는 선언**이다.

## STEP 5 — 진짜 문제 정의

### 1. 표면 문제 정의 (잘못된 정의)

> **4×4 칸에 1부터 16까지 숫자를 넣어, 가로·세로·대각선 합이 같은 마방진을 만드는 프로그램을 작성한다.**

| 함정 | 설명 |
|------|------|
| 목표가 “그림” | 특정 한 답과 성공이 동일시됨 |
| “만든다”만 강조 | 판정 책임 숨김 |
| 완성만 언급 | 부분 상태·실패·여러 해 미포함 |
| TDD·불변식 부재 | 맞다/틀리다가 감에 의존 |

### 2. 개선된 문제 정의 (정확한 정의)

> **4×4 격자에 1~16을 각각 한 번씩 배치하는 배치(완성·부분 모두)에 대해, 마방진 불변식을 만족하는지 여부를 일관된 규칙으로 판정하고, 필요 시 조건을 만족하는 완성 배치를 얻을 수 있는 절차를 재현 가능하게 유지한다. “성공”은 특정 한 장의 격자와 일치가 아니라 불변식 충족으로 정의하며, 판정·생성·표현의 책임을 분리해 변경 시에도 의미가 흔들리지 않게 한다.**

| 요소 | 의미 |
|------|------|
| 대상 | 4×4, 1~16, 유일·범위 |
| 1차 성과 | 불변식에 따른 판정 (만족 / 위반 / 미완) |
| 2차 성과(선택) | 유효 완성 배치를 얻는 절차 |
| 성공 기준 | 임의의 유효 완성 배치 |
| 품질 | 동일 입력·규칙 → 동일 판정·결과 |
| 설계 태도 | 판정 규칙 선고정, 구현은 계약 충족 (TDD) |

### 3. 핵심 Invariant

#### 값·칸

| ID | 불변식 |
|----|--------|
| I1 | 각 칸 값 ∈ {1,…,16} (또는 미배치 모델에 따른 상태) |
| I2 | 완성: 16칸 채움, 1~16 각 정확히 1회 |
| I3 | 부분: 이미 배치된 값 — 중복·범위 위반 없음 |

#### 합 (완성 상태, 마법상수 34)

| ID | 불변식 |
|----|--------|
| I4 | 네 행 합 = 34 |
| I5 | 네 열 합 = 34 |
| I6 | 주·부대각선 합 = 34 |

#### 판정·메타

| ID | 불변식 |
|----|--------|
| I7 | 완성 + I2 + I4~I6 → 유효한 마방진 |
| I8 | I2~I6 위반 → 무효 (위반 종류 구분 권장) |
| I9 | I3 만족·미완성 → 미완 / 판정 보류 |
| I10 | 검증 절차는 생성 절차를 신뢰하지 않음 |
| I11 | 동일 입력 → 동일 판정 |
| I12 | 대표 유효·무효 예는 규칙 불변 시 계속 동일 판정 (회귀 계약) |

### 4. 실제로 훈련하려는 사고 능력

| 능력 | 내용 |
|------|------|
| **불변식 사고** | “합이 같다”를 검사 가능한 조건 목록으로 분해; 완성 vs 부분 구분 |
| **판정권·경계 사고** | 성공 = 유효성; 생성 vs 검증 분리; 위반 유형 구분 |
| **계약·입출력 사고** | Given/Then 고정; 모호한 “만들어 줘” 회피 |
| **재현·회귀 사고** | 반복 실행 전제; 규칙 변경 시 깨질 대표 예 상상 |
| **규칙 명시화** | 손 확인을 절차로 옮길 수준의 규칙 드러내기 |
| **문제 재정의** | “마방진 프로그램” → “불변식 + 판정 + (선택) 완성 + 재현” |
